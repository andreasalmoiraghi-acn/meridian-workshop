Low-Level Design — Inventory Management Dashboard

Meridian Components · RFP #MC-2026-0417 · Based on codebase analysis April 2026

System overview

The Meridian inventory dashboard is a two-tier web application: a Vue 3 single-page application on the frontend and a Python FastAPI REST API on the backend. Data is stored as JSON files loaded into memory at server startup — no relational database. The system supports multi-warehouse operations (San Francisco, London, Tokyo) and dual-language UI (English and Japanese).

Filter logic

apply_filters(items, warehouse?, category?, status?) — sequential filter: warehouse → category (case-insensitive) → status.

filter_by_month(items, month?) — supports YYYY-MM and quarter format Q1-2025…Q4-2025. Quarter map hardcoded.

Reports endpoints (/api/reports/quarterly, /api/reports/monthly-trends) have no filter parameters — always return full-year aggregates regardless of FilterBar state. Fix: R1.

Cross-cutting concerns

Filter flow (end-to-end)

→
useFilters() ref updates
→
watch() triggers loadData()
→
getCurrentFilters()
→
api.js URLSearchParams
→
FastAPI query params
→
apply_filters()

Component re-renders

Filters are a singleton (module-level refs). All views share the same state — changing the warehouse filter in one view affects every view simultaneously on reload.

Reports.vue is the only view that breaks this pattern: it ignores useFilters() and fetches unconditionally. Fix: R1.

i18n flow

LanguageSwitcher click
→
setLocale('ja')
→
localStorage persist
→
currentLocale ref updates
→
t() returns JA strings
→
All components re-render

Translation lookup: current locale → English fallback → return key as-is. Currency auto-switches: en→USD, ja→JPY (×150 conversion hardcoded).

Authentication

Mock only. isAuthenticated is always true. No API calls for login/logout. No route guards. Not a concern for this engagement (internal tool, not internet-facing).

Error handling

Each view manages its own loading, error, and data refs. No global error interceptor. Pattern: try/catch in loadData(), display v-if="error" in template. Network errors show raw message strings.

R2 — Restocking recommendations

FE

New view: Restocking.vue

Budget ceiling input (numeric, required). Recommendation table: SKU, name, warehouse, current stock, recommended qty, estimated cost, priority badge. Filter by warehouse. Integrates useFilters() and useI18n() from day one.

FE

New component: RestockingDetailModal.vue

Detail view for a single recommendation — breakdown by SKU, demand forecast reference, supplier suggestion if available.

FE

New api.js method + router entry

Add getRestockingRecommendations(budget, warehouse). Add route /restocking to main.js. Add nav link in App.vue.

BE

New endpoint: GET /api/restocking

Query params: budget (required), warehouse?. Logic: join inventory + demand_forecasts + backlog_items. Compute recommended_qty = max(0, reorder_point × 1.5 − quantity_on_hand). Filter to items within budget (greedy by priority). Return sorted list. New Pydantic model: RestockingRecommendation.

BE

Populate purchase_orders.json mock data

Add representative PO records so the backlog PO flow is testable end-to-end.

Add CSS custom properties for theme variables (--bg, --surface, --text, etc.) to App.vue. Toggle via data-theme="dark" on <html>. Persist preference to localStorage per workstation. Developed on isolated branch — no risk to main until approved. Zero structural changes to components.
